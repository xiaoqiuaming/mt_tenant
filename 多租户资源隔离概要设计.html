YaoBase多租户资源隔离概要设计

1. 系统概述

1.1 设计目标

在YaoBase分布式数据库系统中实现租户级资源隔离，保障多租户共享集群环境下的资源公平性和SLA保证。支持CPU、内存、磁盘资源的精细化控制和监控。

1.2 核心特性

  • CPU隔离：基于线程池和Linux cgroup v1的双模式CPU资源控制
  • 内存隔离：SqlServer层配额管理，防止单租户过度占用
  • 磁盘隔离：DataServer层存储空间总量控制
  • 资源监控：统一的资源统计接口和实时监控能力

1.3 设计原则

原则 说明
最小侵入 与现有YaoBase架构兼容，减少改造成本
渐进式 先实现基础隔离，后续可扩展高级功能
租户感知 全链路传递租户上下文，实现精确隔离
可配置性 灵活的资源配额配置，满足不同场景需求
可观测性 提供完善的监控和告警机制

3. 核心设计详解

3.1 CPU隔离详细设计

3.1.1 双模式CPU隔离

基础模式（默认）： - 根据租户CPU配额比例分配工作线程 - 全局线程池120个工作线程，按配额比例划分 - 租户A（2核）→ 2/总配额 × 120个线程 - 每个租户独立无锁任务队列，避免竞争

cgroup模式（高级）： - 基于Linux cgroup v1的cpu子系统 - 设置cpu.shares实现份额控制 - 层次结构：/sys/fs/cgroup/cpu/yaobase/tenant_<id>/ - 动态管理线程PID加入/移出cgroup

3.1.2 请求处理流程

客户端连接(user@tenant)
    ↓
TenantAuthenticator.authenticate()
    ↓
CpuQuotaChecker.checkQuota()
    ↓
ThreadPoolManager.submitTask()
    ↓
租户队列入队 → 唤醒工作线程
    ↓
WorkerThread从队列取任务执行
    ↓
CpuMonitor统计资源使用
    ↓
返回结果，释放资源

3.1.3 配额超限处理

限制类型 阈值 处理策略
软限制 80% 记录告警，继续处理，降低优先级
硬限制 100% 拒绝新请求，返回配额不足错误
优雅降级 90% 限流低优先级请求，保障核心业务

3.2 内存隔离设计

3.2.1 内存配额管理

  • 配额检查点：连接建立、大对象分配、执行计划生成
  • 内存统计：实时跟踪租户内存使用（连接缓冲、查询临时数据）
  • 隔离范围：仅SqlServer，TransServer的MemTable为共享资源
  • 释放策略：连接关闭时自动释放，定期清理过期对象

3.2.2 内存使用统计

class MemoryQuotaChecker {
    bool checkMemoryQuota(TenantContext* tenant, size_t requestSize) {
        size_t currentUsage = tenant->getCurrentMemoryUsage();
        size_t quota = tenant->getMemoryQuota();
        if (currentUsage + requestSize > quota) {
            return false; // 拒绝分配
        }
        return true;
    }
};

3.3 磁盘隔离设计

3.3.1 磁盘配额控制

  • 控制层级：DataServer层L1基线数据
  • 统计方式：累加租户所有表的磁盘占用
  • 检查时机：数据合并（Minor/Major Compaction）后
  • 超限处理：禁止新的写入操作，允许读取和删除

3.3.2 延迟性说明

由于LSM-tree架构特性，磁盘统计存在延迟： - 写入先进MemTable（TransServer），不占磁盘 - 合并到DataServer后才计入磁盘使用 - 建议：设置90%预警阈值，提前告警

4. 关键技术实现

4.1 无锁任务队列

采用CAS（Compare-And-Swap）实现高性能无锁队列：

class LockFreeQueue {
    std::atomic<Node*> head;
    std::atomic<Node*> tail;
    bool enqueue(Task* task);   // 无锁入队
    Task* dequeue();            // 无锁出队
};

优势：避免锁竞争，提升并发性能，减少上下文切换。

4.2 cgroup v1集成

# cgroup层次结构
/sys/fs/cgroup/cpu/yaobase/
├── tenant_a/  (cpu.shares=2048, 2核)
├── tenant_b/  (cpu.shares=4096, 4核)
└── system/    (cpu.shares=1024, 系统保留)

实现要点： - 初始化时创建租户cgroup目录 - 设置cpu.shares = 配额 × 1024 - 将工作线程PID写入tasks文件 - 从cpuacct.usage读取CPU使用统计

5. 部署与配置

5.1 配置文件示例

[tenant_a]
cpu = 2.0               # CPU配额（核心数）
memory = 4096           # 内存配额（MB）
disk = 100              # 磁盘配额（GB）
enable_cgroup = true    # 启用cgroup
cpu_soft_limit = 80     # CPU软限制阈值（%）
cpu_hard_limit = 100    # CPU硬限制阈值（%）

[system]
thread_pool_size = 120  # 全局线程池大小
monitor_interval = 5    # 监控采集间隔（秒）

5.2 性能指标

  • 隔离开销：< 5%（相比无隔离）
  • 吞吐量：单租户无明显下降
  • 延迟：P99延迟增加 < 10ms
  • 可扩展性：支持1000+租户

5.3 约束与限制

限制项 说明
磁盘延迟 合并后才检测超限，不实时
TransServer共享 MemTable不隔离，依赖自动合并
数据隔离 共享RAFT架构和LSM-tree结构
cgroup依赖 Linux环境才支持精确CPU控制

5.4 运维建议

  1. 资源规划：配额总和 ≤ 物理资源×0.8，留20%缓冲
  2. 监控告警：配置软限制告警（80%），提前扩容
  3. 定期清理：归档历史数据，释放磁盘空间
  4. 性能测试：上线前验证隔离效果和性能开销
  5. 降级预案：cgroup失败自动降级到基础模式

6. 总结

YaoBase多租户资源隔离系统采用分层隔离、渐进实现的策略，在SqlServer层面实现CPU和内存隔离，在DataServer层面实现磁盘隔离，通过租户上下文全链路传递实现精确的资源控制。系统具备良好的可扩展性和可观测性，能够满足SaaS场景下多租户共享集群的资源管理需求。

核心优势： - ✅ 最小侵入现有架构，改造成本低 - ✅ 支持基础和高级双模式，适应不同场景 - ✅ 完善的监控和告警机制 - ✅ 全面的GoogleTest测试覆盖（90+用例）

后续演进方向： - 支持cgroup v2（更强大的资源控制） - 实现网络带宽隔离 - 提供动态配额调整能力 - 增强多租户事务隔离能力
